# Prompt 风格指南

本文档定义仓库中 Prompt 的语言风格和写作规范，确保所有 Prompt 保持统一、专业、现代化。

---

## 核心原则

### 1. 清晰优于优雅

```markdown
# ✅ 正确 - 清晰直接
请分析以下代码的性能问题：
- 时间复杂度
- 空间复杂度
- 潜在瓶颈

# ❌ 错误 - 过于文艺
让我们一起探索这段代码的性能世界，看看它隐藏着怎样的秘密...
```

### 2. 具体优于抽象

```markdown
# ✅ 正确 - 具体明确
函数名：calculateSum
输入：整数数组，长度 1-10000
输出：整数和
边界：空数组返回 0

# ❌ 错误 - 过于抽象
请分析函数的性能表现
```

### 3. 结构化优于散乱

```markdown
# ✅ 正确 - 结构清晰
输出格式：
{
  "summary": "一句话总结",
  "issues": [
    {
      "severity": "high|medium|low",
      "description": "问题描述"
    }
  ]
}

# ❌ 错误 - 随意描述
输出总结和一些问题，格式随意
```

## 语气指南

### 推荐语气

| 语气 | 使用场景 | 示例 |
|------|----------|------|
| **指令式** | 任务明确时 | "分析代码"、"生成报告" |
| **描述式** | 说明背景时 | "这个 Prompt 用于..." |
| **专业但友好** | 日常沟通 | "请注意..."、"建议..." |

### 语气强度

**过于强硬（避免）**：
```markdown
你必须/一定/绝对要...
```

**推荐（适度）**：
```markdown
请... / 请注意... / 建议...
```

**过于软弱（避免）**：
```markdown
如果你想的话，可以考虑...（太随意）
```

## 句子风格

### 1. 使用主动语态

```markdown
# ✅ 正确
AI 生成代码
用户提供了需求

# ❌ 错误
代码由 AI 生成
需求被用户提供
```

### 2. 使用动词开头

```markdown
# ✅ 正确
分析以下代码...
生成一份报告...
验证修复结果...

# ❌ 错误
这是关于...的分析
报告的生成...
```

### 3. 简洁句子

```markdown
# ✅ 正确
分析错误信息。
定位根因。

# ❌ 错误
首先，我们需要对错误信息进行深入分析。
然后，根据分析结果来确定问题的根本原因。
```

### 4. 避免否定句

```markdown
# ✅ 正确
返回错误信息
使用默认值

# ❌ 错误
不要返回空值
不要使用默认值（双重否定）
```

## 结构化输出

### 何时使用分点

| 场景 | 推荐格式 |
|------|----------|
| 多个步骤 | 编号列表 `1. 2. 3.` |
| 多个选项 | 无序列表 `-` |
| 条件情况 | 分段说明 |
| 层次关系 | 缩进列表 |

### 分点示例

```markdown
请按以下步骤执行：
1. 读取文件内容
2. 分析代码结构
3. 识别关键函数
4. 生成报告

输出包含：
- summary：总结
- details：详情
- suggestions：建议
```

### 表格使用

```markdown
| 字段 | 类型 | 说明 |
|------|------|------|
| name | string | 名称 |
| age | number | 年龄 |
```

## 详细程度平衡

### 足够详细

```markdown
# ✅ 正确 - 详细但不过度
输入：
- error_message（字符串，必填）：完整的错误堆栈
- code_snippet（字符串，可选）：相关代码，最多 500 行

输出：
- root_cause（字符串）：根因分析
- confidence（字符串）：high/medium/low

注意：
- 如果错误信息不完整，返回 error
- 堆栈为空时，尝试从 code_snippet 分析
```

### 过于简略

```markdown
# ❌ 错误 - 不够详细
输入：代码
输出：分析结果
```

### 过于冗长

```markdown
# ❌ 错误 - 过度详细
首先，你需要准备你的输入，这可能包括...
其次，你需要考虑各种边界情况，例如...
最后，你需要确保输出格式正确...
```

## 适配多数模型

### 通用设计原则

1. **指令明确**：不依赖模型推理
2. **格式稳定**：JSON 结构保持一致
3. **上下文清晰**：不假设模型知道背景
4. **容错性好**：处理异常输入

### 模型适配建议

```markdown
# ✅ 兼容性好
输出 JSON 格式：
{
  "result": "结果",
  "status": "success|error"
}

# ❌ 可能有问题
输出"最好的结果"（不明确）
```

## 避免的问题写法

### 1. 模糊指令

```markdown
# ❌ 错误
"好好分析一下代码"

# ✅ 正确
"分析代码的正确性、错误处理、性能表现"
```

### 2. 矛盾要求

```markdown
# ❌ 错误
"输出简短但详细的报告"

# ✅ 正确
"输出一句话总结 + 详细分析（可选字段）"
```

### 3. 假设过多

```markdown
# ❌ 错误
"用你熟悉的方式处理"

# ✅ 正确
"使用 JSON 格式输出，UTF-8 编码"
```

### 4. 冗余表述

```markdown
# ❌ 错误
"首先，第一步要做的第一件事是..."

# ✅ 正确
"第一步：..."
```

### 5. 不一致格式

```markdown
# ❌ 错误
输出：
- name: 张三
- age: 20

# ✅ 正确
输出：
{
  "name": "张三",
  "age": 20
}
```

## 变量表示

### 变量格式

```markdown
# 标准变量
${variable_name}

# 带类型的变量（可选）
${variable_name:string}

# 默认值变量
${variable_name:default_value}
```

### 变量命名

```markdown
# ✅ 正确
${error_message}
${code_snippet}
${file_path}

# ❌ 错误
${msg} (缩写过于简单)
${the_code} (描述不清)
${input1} (无意义名称)
```

## 段落结构

### 好的结构

```markdown
## Context
[使用背景]

## Prompt Body
[核心指令]

## Variables
[变量定义]

## Usage
[使用方式]

## Example
[示例]
```

### 避免的结构

```markdown
# ❌ 大段文字
## Context
这是用于...的 Prompt，它可以帮助你...当你需要...的时候可以使用这个 Prompt...

# ✅ 分段清晰
## Context
用于代码审查场景。
在完成代码开发后使用。
```

## 检查问题

写完后自问：

1. **清晰度**：别人能一眼看懂这个 Prompt 做什么吗？
2. **完整性**：所有必要信息都包含了吗？
3. **可执行性**：AI 能直接按这个 Prompt 执行吗？
4. **格式一致性**：与仓库其他 Prompt 风格一致吗？
5. **详细程度**：不过多也不过少吗？
